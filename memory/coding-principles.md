# Coding Principles

綴 TSUZURI / 茶帳における実装判断基準。

コードは、ただ動けばよいものではない。

長く育てられること。
あとから読み返せること。
世界観を壊さないこと。
利用者の所作を邪魔しないこと。

この4つを重視する。

---

# 基本姿勢

## 推測で修正しない

原因が分からないまま修正しない。

関連ファイルを確認し、
データの流れ、責務、依存関係を把握してから変更する。

「たぶんこれで直る」は避ける。

---

## 最小変更を優先する

一度に大きく壊さない。

必要な範囲だけを丁寧に直す。

ただし、根本原因が設計にある場合は、
対症療法ではなく構造改善を提案する。

---

## 動けばいいで終わらせない

動作確認は最低ライン。

そのうえで、

- 読みやすいか
- 責務が分かれているか
- 型が安全か
- 将来修正しやすいか
- UIの思想と合っているか

を確認する。

---

# 設計思想

## 小さく作る

巨大なファイルを避ける。

1つのファイルに、
表示、状態管理、API通信、整形処理、バリデーションを詰め込まない。

---

## 責務を分ける

役割を明確にする。

- View：表示とユーザー操作
- composables：状態と振る舞い
- API層：通信
- types：型定義
- utils：純粋な整形・変換処理

責務が混ざったら分離を検討する。

---

## 長期保守を前提にする

今だけ動くコードではなく、
半年後の自分が読めるコードにする。

未来の修正者が迷わないことを重視する。

---

# Vue / TypeScript 方針

## Vue 3 Composition API を前提にする

Composition API を使い、
状態とロジックを整理する。

Viewにロジックを溜め込みすぎない。

---

## composables の責務

composables は、
再利用可能な状態・処理をまとめる場所。

ただし、
何でも入れる場所ではない。

### 良い composable

- useTeaUtensils
- useKeikoLogs
- useAuthUser
- useDialogState
- useImageUpload

### 避ける composable

- useEverything
- usePageLogic
- useCommon
- useUtils

名前から責務が分かること。

---

## Viewを肥大化させない

Viewファイルは、
画面構成が読み取れる状態を保つ。

複雑な処理は composables / utils / API層へ逃がす。

---

## props / emits を明確にする

親子間の責務を曖昧にしない。

子コンポーネントは、
親の状態を直接知りすぎない。

イベント名は具体的にする。

---

# 型安全

## any を安易に使わない

`any` は最後の手段。

型が分からない場合は、
まずデータ構造を確認する。

必要なら型を定義する。

---

## 型逃げをしない

エラーを消すためだけの型アサーションを避ける。

悪い例：

```ts
const item = data as any;
```

より良い例：

```ts
type TeaUtensil = {
  id: string;
  name: string;
  category: string | null;
};
```

---

## null / undefined を明示する

存在しない可能性を曖昧にしない。

Supabase 由来のデータは、
null を含む可能性を考慮する。

---

## 型は思想である

型定義は単なる補助ではなく、
アプリの構造を表すもの。

名前、責務、関係性が伝わる型にする。

---

# Supabase 方針

## RLS前提で考える

認証ユーザーごとのデータ分離を前提にする。

クライアント側だけで守ろうとしない。

---

## クエリを散らばらせない

Supabaseへのアクセスは、
できるだけAPI層または composables に集約する。

同じようなクエリを複数箇所に書かない。

---

## Storage処理は分離する

画像アップロード、圧縮、削除、並び替えなどは、
UIから分離する。

UIは「何が起きているか」を扱い、
詳細処理は専用関数へ移す。

---

# UI実装方針

## 世界観を壊さない

機能追加によって、
茶帳や綴の静けさを壊さない。

便利さより、
自然に使えることを優先する。

---

## 余白を保つ

コンポーネントを詰め込みすぎない。

情報量が増える場合は、
階層化、分割、折りたたみを検討する。

---

## アニメーションは意味がある時だけ

動きで目立たせない。

状態変化を自然に伝えるために使う。

---

## ボタンを増やしすぎない

操作が多すぎると、
利用者の所作が乱れる。

主要操作と補助操作を分ける。

---

# 茶帳固有の実装思想

## 茶道具が主役

茶道具管理画面では、
UIより道具が目立つこと。

装飾が道具を邪魔しないこと。

---

## 記録が続くことを優先する

高機能より、
入力しやすさを優先する。

長文入力を強制しない。

少しだけでも残せる設計にする。

---

## 管理アプリ感を減らす

茶帳は業務システムではない。

茶道の記録と感覚を残す場所。

表や管理画面のような圧迫感を避ける。

---

# レビュー方針

コードレビューでは以下を見る。

1. 責務が分かれているか
2. 型が安全か
3. 重複がないか
4. 命名が自然か
5. UI思想と合っているか
6. 将来の変更に耐えられるか
7. 今回の修正が過剰でないか

---

# 修正提案の出し方

修正するときは、
必ず以下を明示する。

- 何が問題か
- なぜ問題か
- どこを直すか
- どう直すか
- 影響範囲
- 確認すべき動作

---

# 禁止する実装

- 原因未確認の修正
- any乱用
- 巨大View
- 巨大composable
- 責務不明の common / utils
- 同じ処理のコピペ
- 型エラーを黙らせるだけの修正
- UI思想を無視した機能追加
- 一時対応の放置

---

# 良いコードの定義

良いコードとは、
賢そうなコードではない。

静かで、読みやすく、
責務が分かれていて、
あとから直せるコード。

綴 TSUZURI / 茶帳では、
コードにも余白と所作があることを重視する。
